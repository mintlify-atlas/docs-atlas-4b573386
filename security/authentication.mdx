## Overview

Echo's authentication system is designed with privacy and security as core principles. Unlike traditional platforms, Echo never stores your email or password in retrievable form.

## Security architecture

Echo uses a multi-layered approach to protect user credentials:

<Steps>
  <Step title="Cryptographic hashing">
    User credentials are transformed into an irreversible hash:

    ```typescript lib/utils.ts
    import { createHash } from 'crypto';

    export function generateUserHash(
      email: string,
      password: string
    ): string {
      const salt = process.env.AUTH_SALT;
      const combined = `${email}:${password}:${salt}`;
      return createHash('sha256').update(combined).digest('hex');
    }
    ```

    **Security properties**:
    - SHA-256 is a one-way cryptographic hash function
    - Salt prevents rainbow table attacks
    - Combines email + password + salt for uniqueness
    - Output is deterministic (same inputs = same hash)
    - Irreversible (can't recover email/password from hash)
  </Step>

  <Step title="Database storage">
    Only the hash is stored:

    ```prisma
    model User {
      id       String @id @default(cuid())
      userHash String @unique  // Only this is stored
      username String @unique
      // Email and password are NEVER stored
    }
    ```

    **Benefits**:
    - Database breach doesn't expose emails/passwords
    - Staff can't access user credentials
    - Compliance with privacy regulations
  </Step>

  <Step title="Session tokens">
    After authentication, JWT tokens manage sessions:

    ```typescript
    // lib/auth.ts
    export const config = {
      session: {
        strategy: "jwt",
        maxAge: 60 * 60 * 24 * 30, // 30 days
        updateAge: 60 * 60 * 24,   // Refresh every 24h
      },
      secret: process.env.NEXTAUTH_SECRET,
    };
    ```

    Tokens are:
    - Encrypted with `NEXTAUTH_SECRET`
    - Stored as HTTP-only cookies
    - Automatically refreshed
    - Invalidated on logout
  </Step>
</Steps>

    **Security measures**:
    1. Email domain validation prevents fake accounts
    2. Verification code proves email ownership
    3. Password requirements enforce strong passwords
    4. Username uniqueness prevents impersonation
    5. userHash prevents credential exposure
  </Tab>

  <Tab title="Login">

    1. No separate password verification - hash comparison only

## Password requirements

Echo enforces strong password policies:

```typescript lib/utils.ts
export function validatePassword(password: string): boolean {
  // Length: 8-50 characters
  if (password.length < 8 || password.length > 50) {
    return false;
  }

  // Must contain:
  const hasUppercase = /[A-Z]/.test(password);
  const hasLowercase = /[a-z]/.test(password);
  const hasNumber = /[0-9]/.test(password);
  const hasSpecial = /[^A-Za-z0-9]/.test(password);

  return hasUppercase && hasLowercase && hasNumber && hasSpecial;
}
```

**Requirements**:
- ✅ 8-50 characters
- ✅ At least one uppercase letter (A-Z)
- ✅ At least one lowercase letter (a-z)
- ✅ At least one number (0-9)
- ✅ At least one special character (!@#$%^&*)

<Warning>
  These requirements significantly increase password entropy and resistance to brute force attacks.
</Warning>

## Email verification

Echo uses a secure email verification flow:

<CodeGroup>
```typescript Send Code
// POST /api/verifyCollege/send
export async function POST(req: Request) {
  const { email } = await req.json();

  // Generate 15-character random code
  const code = crypto.randomBytes(8).toString('hex').substring(0, 15);

  // Store code in Redis with 15-minute expiration
  await redis.set(
    `verify:${email}`,
    code,
    { ex: 900 } // 15 minutes
  );

  // Send email via Resend
  await resend.emails.send({
    from: 'noreply@echo.app',
    to: email,
    subject: 'Verify your Echo account',
    html: `Your verification code is: <strong>${code}</strong>`,
  });

  return NextResponse.json({ success: true });
}
```

```typescript Verify Code
// POST /api/verifyCollege/verify
export async function POST(req: Request) {
  const { email, code } = await req.json();

  // Retrieve stored code from Redis
  const storedCode = await redis.get(`verify:${email}`);

  if (!storedCode || storedCode !== code) {
    return NextResponse.json(
      { error: 'Invalid verification code' },
      { status: 400 }
    );
  }

  // Delete code after successful verification
  await redis.del(`verify:${email}`);

  return NextResponse.json({ success: true });
}
```
</CodeGroup>

**Security properties**:
- 15-character codes are computationally infeasible to guess
- Codes expire after 15 minutes
- One-time use (deleted after verification)
- Stored in Redis, not database
- Rate limited to prevent spam

## Session management

NextAuth.js handles secure session management:

<Accordion title="Session configuration">
  ```typescript
  export const config = {
    session: {
      strategy: "jwt",
      maxAge: 60 * 60 * 24 * 30,        // 30 days
      updateAge: 60 * 60 * 24,          // Refresh daily
      generateSessionToken: () => {
        return randomBytes(64).toString('hex');
      },
    },
    cookies: {
      sessionToken: {
        name: 'next-auth.session-token',
        options: {
          httpOnly: true,      // Not accessible via JavaScript
          sameSite: 'lax',     // CSRF protection
          path: '/',
          secure: process.env.NODE_ENV === 'production', // HTTPS only in prod
        },
      },
    },
  };
  ```
</Accordion>

<Accordion title="JWT payload">
  ```typescript
  // What's stored in the JWT
  {
    id: "user_cuid",
    username: "johndoe",
    college: "University Name",
    isAdmin: false,
    iat: 1234567890,  // Issued at
    exp: 1237159890,  // Expires at
  }

  // NOT stored:
  // - email
  // - password
  // - userHash
  // - any sensitive data
  ```
</Accordion>

<Accordion title="Session refresh">
  ```typescript
  // Automatic refresh when session is older than updateAge
  callbacks: {
    async jwt({ token, user, trigger, session }) {
      // On sign in
      if (user) {
        token.id = user.id;
        token.username = user.username;
        token.college = user.college;
        token.isAdmin = user.isAdmin;
      }

      // On manual update
      if (trigger === 'update' && session?.user) {
        token.username = session.user.username;
      }

      return token;
    },
  }
  ```
</Accordion>

## Security headers

Recommended security headers for Next.js:

```javascript next.config.js
const securityHeaders = [
  {
    key: 'X-DNS-Prefetch-Control',
    value: 'on'

  {

  },
  {

  {
    key: 'X-XSS-Protection',
    value: '1; mode=block'

    return [
      {
        source: '/(.*)',
        headers: securityHeaders,
      },
    ];
  },
};
```

## Account recovery

<Warning>
  **Important limitation**: Because Echo doesn't store emails, account recovery is not possible if you forget your password.
</Warning>

Users must:
- Remember their email and password
- Use a password manager
- Create a new account if credentials are lost

**Trade-off**: Maximum privacy vs. convenience

If you need account recovery, you could:
1. Store encrypted email addresses
2. Implement password reset via email
3. Use passkey authentication (WebAuthn)

## Compliance considerations

<Accordion title="GDPR compliance">
  Echo's architecture supports GDPR:

  - **Right to erasure**: Delete user record (userHash isn't PII)
  - **Data minimization**: Only stores necessary data
  - **Pseudonymization**: userHash instead of email
  - **Security**: Cryptographic protection

  ```typescript
  // Delete user data
  await db.user.delete({
    where: { id: userId },
    // Cascade deletes posts, likes, etc.
  });
  ```
</Accordion>

<Accordion title="FERPA compliance (US education)">
  Echo doesn't store student records, only anonymous social data:

  - No personally identifiable information
  - No academic records
  - No directory information
  - College affiliation only
</Accordion>
